03 · THE PROBLEM

Why Python slows down on hot loops.

interpreted
Every line is translated on the fly — no ahead-of-time compilation.
dynamic types
Every + checks types, dispatches through objects, updates refcounts.
object overhead
An int is not 8 bytes — it's a heap object with a header.
Analogy
Python is a brilliant translator at a summit — accurate, nuanced, responsive. But translation is per-sentence. For a million sentences in a row, you want a prewritten script.
Narration

Python is interpreted, dynamically typed, and everything — even a plain integer — is a heap object. That flexibility is amazing for writing code, but it's expensive per operation. A million tiny operations in a row, and the overhead dominates. This isn't a Python bug. It's a design trade-off.

04 · THE IDEA

Drop down to C, just for the hot parts.

what a c extension is
A shared library — written in C — that Python can import like any module and call like any function.
python on top → readable c underneath → fast callers never know the difference
best of both worlds
Py × C
Keep Python's expressiveness. Borrow C's speed. Pay the cost only where it matters.
Narration

Here's the idea: what if, just for the hot parts, we could drop down a level — into C — and come right back up? That's a C extension. You write the hot function in C, Python calls it like any other function, and the caller has no idea. Best of both worlds.

06 · FIRST EXAMPLE

A first hotspot — sum_squares(n).

python · sum_squares.py
def sum_squares(n):
    s = 0
    for i in range(n):
        s += i * i
    return s

sum_squares(10_000_000)
# ~900 ms
c · fastmath.c
static PyObject* sum_squares(
    PyObject* self, PyObject* args) {
  long n;
  if (!PyArg_ParseTuple(args, "l", &n))
    return NULL;

  long long s = 0;
  for (long i = 0; i < n; i++)
    s += (long long) i * i;

  return PyLong_FromLongLong(s);
}
// ~12 ms
Narration

Same logic, two languages. The Python version is about 900 milliseconds for 10 million iterations. The C version — same loop, no interpreter overhead — finishes in about 12 milliseconds. Small function, huge payoff. This is exactly the kind of hotspot a C extension is made for.

08 · UNDER THE HOOD

The C API — a small, sharp toolbox.

parse args
PyArg_ParseTuple(args, "l", &n)
Decode what Python sent — ints, strings, objects.
return values
Py_BuildValue("l", result)
Wrap a C value back into a Python object.
refcounts
Py_INCREF / Py_DECREF
Keep the GC honest — you own what you create.
fastmath.c
static PyMethodDef FastMath_Methods[] = {
  {"sum_squares", sum_squares,
   METH_VARARGS, "sum of i*i for i in range(n)"},
  {NULL, NULL, 0, NULL}
};

static PyModuleDef fastmathmodule = {
  PyModuleDef_HEAD_INIT, "fastmath",
  NULL, -1, FastMath_Methods
};

PyMODINIT_FUNC PyInit_fastmath(void) {
  return PyModule_Create(&fastmathmodule);
}
Narration

Python ships with a C API — headers and functions that let C speak Python's language. You parse incoming arguments, build return values, and register your functions in a method table that Python imports as a module. Small set of primitives, but it's all you need.

12 · TRADE-OFFS

Nothing comes free.

● complexity
Two languages, two build systems, two debugging mental models.
● memory
Python's GC doesn't see your C allocations — refcounts are your job.
● debugging
A segfault in C takes down the whole process — no Python traceback.
Go in with eyes open. The speedup is real — so is the maintenance cost.
Narration

Nothing comes free. Three real costs: complexity — you maintain two languages. Memory — Python's garbage collector doesn't know about your C allocations, so you manage reference counts yourself. And debugging — a segfault in C takes down the whole Python process, with no helpful traceback.

13 · ⚠️ COMMON MISTAKES

Four traps every first-timer falls into.

  • 01Skipping Py_INCREF / Py_DECREF — silent leaks today, a crash next Tuesday.
  • 02Returning a borrowed reference as if you owned it — the object gets freed under you.
  • 03Holding the GIL through long C work — you block every other Python thread for the full duration.
  • 04Ignoring NULL returns from C API calls — any of them can fail, and Python expects you to check.
Narration

A few common mistakes to watch for. Forgetting incref and decref causes silent memory leaks and crashes much later. Returning a borrowed reference as if you owned it leads to use-after-free. Holding the GIL through long C work blocks every other thread. And skipping error checks on API calls — any of them can return NULL.

14 · WHEN NOT TO

When not to reach for C.

● i/o-bound work
You're waiting on disk, network, or a database — not burning CPU.
Try async, batching, or caching first.
● no profile yet
You're guessing at what's slow.
Measure first — the hotspot is never where you think.
● rarely called
A 200 ms function that runs once a day needs no rewrite.
Readability wins in cold code paths.
Narration

When should you not reach for C? When you're I/O-bound — waiting on disk, network, or a database. When you haven't profiled yet and you're guessing. And when the code runs once a day. A 200-millisecond function doesn't need a C rewrite, no matter how satisfying it would feel.

16 · ALTERNATIVES

The raw C API isn't the only door.

cython
Python-like syntax that compiles to C. Fewest surprises, gentlest ramp.
cffi · ctypes
Call an existing C library from Python without writing extension code.
pyo3 (rust)
Extensions in Rust with memory safety, modern tooling, and great ergonomics.
numba
JIT numerical Python with a decorator — no build step, no extra language.
Pick the tool that matches your team's skills and your performance budget. The C API is the foundation — it's rarely the most ergonomic choice.
Narration

The raw C API isn't your only option. Cython gives you Python-like syntax that compiles to C. CFFI and ctypes call existing C libraries without extension code. PyO3 brings Rust and memory safety. Numba JITs numerical code with a decorator. Pick the tool that matches your team and your budget.

17 · KEY TAKEAWAYS

Five things to carry home.

  • 01C extensions are a scalpel, not a hammer — use them on hotspots.
  • 02Profile before you cut. Measured beats guessed, every time.
  • 03Respect the costs — complexity, memory, debugging are real.
  • 04Start higher-level — Cython, NumPy, PyO3 often get you 90% of the win.
  • 05Know the four-step loop — write, declare, compile, import.
Narration

Let's bring it home. C extensions are a scalpel, not a hammer. Profile before you cut. Target hotspots, not whole systems. Respect the trade-offs. Start with higher-level tools when you can. And remember the four-step loop — write, declare, compile, import. That's the workshop.
